# Tools Guide

这个目录放 MCP `tool`，也就是会被客户端显式调用的可执行能力。

## 一个 tool 文件应该长什么样

最低限度通常包含两部分：

1. 具体函数，实现能力本身
2. `register_*_tools(mcp)`，负责向 `FastMCP` 注册

## 设计时优先关注

- 入参是否明确
- 返回值是否稳定
- 是否容易单测
- 是否把业务逻辑和 MCP 边界混在了一起

## 推荐工作流

### 当你要新增一个简单 tool

1. 新建按领域命名的模块，比如 `filesystem.py`、`weather.py`
2. 先写纯函数或薄业务函数
3. 再补注册函数
4. 最后在 `tools/__init__.py` 中统一接入

### 当一个 tool 变复杂

出现下面任意一种情况，就不要继续把逻辑塞在这里：

- 同一模块里开始出现多段外部 API 调用
- 需要共享数据结构
- 需要重试、鉴权、缓存、格式转换

这时优先把逻辑下沉到未来的 `services/` 或 `adapters/`。

## 不建议

- 在 tool 函数里直接读取环境变量
- 在 tool 模块里混放 unrelated 工具
- 为了“以后可能复用”过早创建通用大而全工具箱
